# PGM-00009 — pgm-codeit: fan out over NEEDS ADDRESS pool

- **Slice value:** one command (`/pgm-codeit`) drives both "build new approved work" and "address open review comments" in the same fan-out proposal, instead of requiring a separate manual invocation of `pgm-address-comments` per flagged ticket.

## Intent
`/pgm-codeit`'s step 2 already reads `board.py ready`; once PGM-00007 adds the NEEDS ADDRESS
bucket to that output, extend the step-3 announcement to list **both** pools separately (READY
tasks with the normal start→build→review→PR pipeline; NEEDS ADDRESS tickets running
`pgm-address-comments`, PGM-00004) with counts, and let a cap apply per-pool. Default (no args)
fans out over everything in both pools, same announce-and-confirm gate as today.

For NEEDS ADDRESS tickets specifically: worktree is **not** created via `board.py wt` (that
does an `Approved -> In Progress` transition, which is invalid here — ticket is `In Review`).
Instead do a plain `git worktree add <dir> <branch>` on the ticket's existing branch (already
exists — same `branch_name(fm)` formula), run `pgm-address-comments <id>` there, then plain
`git worktree remove` when done. No `board.py wt rm` either.

**Parked** — depends on PGM-00007 which is itself deferred; this is the automation layer on top
of manual `pgm-address-comments` (PGM-00004), revisit once that's proven out.

## Acceptance criteria
- [ ] `/pgm-codeit`'s announcement lists READY and NEEDS ADDRESS as two separate counted pools.
- [ ] A cap (count / `parallel N`) can be applied per-pool.
- [ ] NEEDS ADDRESS tickets get a worktree via plain `git worktree add` on their existing branch, not `board.py wt`.
- [ ] Each NEEDS ADDRESS worktree runs `pgm-address-comments <id>`, then is torn down via plain `git worktree remove`.
- [ ] No status transition attempted for NEEDS ADDRESS tickets by pgm-codeit itself (the skill handles `unflag`).
